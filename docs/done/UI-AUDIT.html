App Shell & Layout

+layout.svelte · app.css

⚠ Advertencia

LAYOUT-01

Bottom nav ausente en /session — pero SessionBar comparte z-index

La ruta /session no muestra el bottom nav del layout, lo cual es correcto — el DM panel necesita toda la pantalla. Sin embargo, tanto el bottom nav (cuando aparece) como .sb-bar usan z-index: var(--z-nav). Si se navega a /session mientras el sidebar está abierto, el sidebar backdrop (z:30) queda por debajo de SessionBar (z:50) correctamente, pero la transición puede verse cortada. No es crash, pero puede confundir al DM.
Fix Definir --z-session-bar: 45 en tokens para SessionBar — por debajo del sidebar (50) pero por encima del contenido. Verificar que el sidebar se cierre al navegar a /session.
✓ OK

LAYOUT-02

Header fixo con brand wordmark y estado de conexión

El header muestra correctamente el dot de conexión Socket.io, el contador de personajes, y el título de página dinámico. El botón ☰ tiene aria-label="Abrir navegación". La implementación está completa y consistente con el design system.
⚠ Advertencia

LAYOUT-03

Sidebar no tiene trap de foco (P1 conocido)

El Modal tampoco tiene focus trap — esto está en el backlog (P1 en DESIGN-SYSTEM.md). Con el sidebar abierto, Tab navega a elementos debajo del overlay. En el contexto de una sesión de juego (DM en el teléfono) el riesgo es bajo, pero viola WCAG 2.4.3.
Fix Usar inert en #main-content cuando el sidebar está abierto — o un trap de foco manual con focus-trap-svelte. Aplicar también a Modal.
⚔

InitiativeStrip

InitiativeStrip.svelte · InitiativeStrip.css

✗ Crítico

IS-CRIT-01

Nombre del token a 0.65rem (~10.4px) — ilegible en condiciones de juego

.is-token-name tiene font-size: 0.65rem = ~10.4px a 16px base. En el modo combate, este texto muestra el primer nombre del personaje (ej. "MARCELO") en un espacio de 64–80px. El DM sostiene el teléfono a ~40–60cm durante una sesión activa. A esa distancia, 10px en pantalla de alta densidad equivale a ~0.18mm visual — debajo del umbral de lectura confortable (0.2mm recomendado ANSI).
Fix Subir a font-size: 0.7rem (11.2px). Reducir el máx de caracteres visibles usando max-width: 100% y overflow: hidden; text-overflow: ellipsis (ya implementado). Alternativamente mostrar solo iniciales cuando el token tiene < 72px de ancho.
✗ Crítico

IS-CRIT-02

Roll number a 0.75rem en token inactivo — contraste insufiiciente (gris sobre negro)

.is-token-roll usa color: var(--grey) (#888888) sobre var(--black-card) (#111111). Ratio calculado: 5.4:1. Para texto de 12px (no-bold) el mínimo WCAG AA es 4.5:1 — pasa AA pero falla en la práctica porque es una etiqueta funcional (el número de iniciativa es crítico para el DM) y el ambiente de uso es un espacio con luz variable (mesa de juego, pantalla brillante).
Fix Usar color: rgba(240,240,240,0.6) (~7:1) en tokens inactivos para el número. El estado activo ya usa var(--cyan) (~12:1) correctamente.
⚠ Advertencia

IS-01

max-height: 260px en lista de rolls puede ocultar personajes en teléfonos pequeños

Con 4 personajes y min-height: 52px por row + gaps, la lista mide ~228px — entra justo. Pero con 5+ personajes en campañas futuras, el contenido se desbordará y habrá scroll invisible. El usuario no recibe señal visual de que hay más elementos debajo del corte.
Fix Añadir mask-image: linear-gradient(to bottom, black 80%, transparent 100%) al scroll container cuando hay overflow. O mostrar un contador "X más abajo ↓" como texto de ayuda.
✓ OK

IS-02

Roll input: 44px height ✓ — WCAG 2.5.5 touch target correcto

Comentario en código confirma que fue elevado de 36px → 44px deliberadamente. El input de 52px ancho también es suficiente para dígitos d20 (-5 a 30). Correcto.
✓ OK

IS-03

Token activo: glow + indicador pulsante + color de nombre y roll — diferenciación clara

El token activo usa 3 señales visuales independientes: borde cyan + doble box-shadow glow, punto indicador pulsante abajo, y cambio de color en nombre (gris → blanco) y roll (gris → cyan). Correcta redundancia para no depender de color solo.
👤

SessionCard

SessionCard.svelte · SessionCard.css

✗ Crítico

SC-CRIT-01

Condition badges a 0.65rem — etiquetas funcionales demasiado pequeñas

.sc-condition-badge usa font-size: 0.65rem. Aunque el comentario /* SC-2: raised from 0.6rem */ muestra consciencia del problema, 0.65rem (~10.4px) sigue siendo insuficiente para etiquetas que el DM debe leer en tabla durante una sesión ("ENVENENADO", "ASUSTADO"). Las condiciones son información crítica de combate.
Fix Subir a font-size: 0.75rem (12px) y ajustar padding a 2px 6px para mantener la proporcionalidad. Las pills se harán ligeramente más anchas pero los cards tienen ancho suficiente (≥280px). Verificar que no rompa el layout en cards de 280px de ancho con 3+ condiciones activas.
⚠ Advertencia

SC-01

Foto de 56px puede mostrar silueta gris si la imagen no carga

resolvePhotoSrc() devuelve un fallback si photo es null, pero no hay onerror en el <img> que cambie a un placeholder visual. Si el servidor no está disponible (WiFi cortado durante sesión) las fotos se verán rotas con el ícono de imagen rota del browser.
Fix Añadir onerror="this.src='/placeholder-avatar.png'" al <img class="sc-photo">. O usar un avatar generado con las iniciales del personaje como fallback CSS puro.
⚠ Advertencia

SC-02

HP bar de 4px es muy delgada para percepción periférica

La barra de 4px es suficiente para un dashboard analítico, pero en el DM panel el estado de HP se lee en un vistazo rápido. El CharacterCard en /control/characters usa una barra más prominente. En condición crítica la barra pulsa pero el área de animación es mínima.
Fix Subir .sc-hp-track a height: 6px — sigue siendo compacto pero mejora la percepción visual del estado de HP de un vistazo. Verificar animación de pulso en estado crítico sigue siendo suave.
✓ OK

SC-03

Grid minmax(280px, 1fr) — comportamiento correcto en todos los anchos

En 320px → 1 columna de 280px. En 375px → 1 columna de 311px. En 768px → 2 columnas. En 1024px → 3 columnas. La distribución es apropiada para todos los dispositivos objetivo. Ningún card queda demasiado estrecho o demasiado ancho.
✓ OK

SC-04

Resource pips (10px) — decorativos, no interactivos — tamaño aceptable

Los pips son aria-hidden="true" y el grupo tiene aria-label con el valor textual. No son targets táctiles — solo visualizan información. 10px es suficiente como indicador visual. Correcto.
ℹ Info

SC-05

Estado is-selected y is-active simultáneos — cyan gana sobre rojo crítico

Cuando una carta está en crítico (rojo) y el DM la selecciona (is-selected), el borde cambia a cyan. Esto es correcto — la selección activa es más importante que el estado de HP en el flujo de acción. Sin embargo, un usuario nuevo puede interpretar el cambio de rojo a cyan como que el personaje dejó de estar crítico. Considerar mantener un glow rojo secundario o un indicador de "⚠" visible aunque la carta esté seleccionada.
🎮

SessionBar

SessionBar.svelte · SessionBar.css

✗ Crítico

SB-CRIT-01

Action buttons horizontalmente scrolleables sin señal visual de overflow

En un teléfono de 360px, los 4 action buttons (⚔ Daño · ✦ Curar · ◈ Condición · ☽ Descanso) más el botón Siguiente ▶ tienen un ancho total estimado de ~460px — más ancho que el viewport. El contenedor .sb-actions tiene overflow-x: auto; scrollbar-width: none — el scroll está disponible pero completamente invisible. Un DM nuevo probablemente no descubrirá "Descanso" o el botón Siguiente. Esto es un flujo crítico del DM panel.
Fix opción A Añadir mask-image: linear-gradient(to right, black 70%, transparent 100%) al .sb-actions para mostrar un fade que indique overflow. Es CSS puro, no rompe nada.
Fix opción B Hacer el layout de 2 filas en <480px: primera fila con ⚔ ✦ | ☽, segunda fila con ◈ + ▶ Siguiente. Esto elimina el scroll pero aumenta la altura del bar a ~110px.
⚠ Advertencia

SB-01

Input numérico de daño/cura tiene max-width: 80px — puede quedar tapado por teclado virtual

En estado de confirmación, el teclado numérico ocupa ~250px de altura en iOS/Android. La SessionBar está al bottom con position: fixed. Cuando el teclado sube, el bar queda por debajo del teclado — pero el input puede quedar oculto parcialmente si el viewport no hace scroll. Esto afecta el flujo de daño/cura, que son las acciones más frecuentes.
Fix Añadir inputmode="numeric" al input de número en lugar de type="number" (evita flechas del browser). Añadir un $effect que haga scrollIntoView() en el bar cuando el input recibe foco. Alternativamente, considerar un número pad nativo en la UI en lugar del input de teclado.
⚠ Advertencia

SB-02

Fuente 0.75rem en botones — al límite del mínimo legible con guantes o dedos gordos

.sb-btn tiene font-size: 0.75rem (12px). Con letter-spacing: 0.06em y mayúsculas, el espacio efectivo entre letras es ok, pero el tamaño de ~12px es el mínimo absoluto para fuentes de display como var(--font-display). En DMs con dedos anchos el texto puede verse pequeño mientras el área táctil (44px) es suficiente.
Fix Subir a font-size: 0.8rem (12.8px). Revisar que todos los botones mantengan su label en una sola línea — si hay desbordamiento, quitar el emoji de los labels (⚔ Daño → solo Daño).
✓ OK

SB-03

3-state transition con animación fade-in — excelente UX para flujo de 2 taps

La transición idle → pending → confirm con @keyframes sbFadeIn (opacity 0→1, translateY 4px→0) es sutil pero efectiva. El DM ve claramente en qué estado está. El border-top animado (gris → gris-claro → cyan) es una señal visual secundaria bien ejecutada.
✓ OK

SB-04

Botón confirm cambia a rojo para "Agregar condición" — diferenciación semántica correcta

.sb-btn--confirm-add usa rojo para el botón de agregar condición (una acción potencialmente negativa para el personaje) mientras que daño/cura usan cyan. La distinción semántica refuerza la naturaleza de la acción y reduce errores de DM en sesión.
🃏

CharacterCard

CharacterCard.svelte · CharacterCard.css

⚠ Advertencia

CC-01

.char-card.collapsed debería ser .is-collapsed (BEM convention)

Documentado en DESIGN-SYSTEM.md como "known issue". La convención del proyecto es .is- prefix para todos los estados booleanos. El outlier .collapsed puede confundir a futuros devs y quebrarse si se usa .is-collapsed en algún componente nuevo.
Fix Renombrar en CharacterCard.svelte de class:collapsed a class:is-collapsed y actualizar CharacterCard.css. Cambio de 1 línea en cada archivo, cero riesgo.
✓ OK

CC-02

Drag-to-reorder implementado con drag handle explícito — excelente UX

El drag handle de 20px sobre el card previene reorders accidentales al hacer scroll. El wrapper .char-card-exit-wrap.is-dragging eleva visualmente la carta durante el drag. Patrón correcto.
ℹ Info

CC-03

anime.js importado desde dos paths diferentes entre componentes

Documentado en DESIGN-SYSTEM.md (P2). Puede causar un doble-bundle de la librería en el build. No es visible en UX pero afecta performance de carga (bundle size). Verificar con bun run build && npx vite-bundle-analyzer.
