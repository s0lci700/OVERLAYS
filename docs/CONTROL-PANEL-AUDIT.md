# Control Panel Architecture & Styling Audit
> **Repo:** OVERLAYS (TableRelay) — `control-panel/`
> **Date:** 2026-03-09
> **Scope:** Svelte 5 control-panel — tokens, components, styling, UI library usage

---

## 1. High-Level Styling Architecture

The control panel uses a **four-layer hybrid CSS system**:

| Layer | Files | Role |
|---|---|---|
| Design tokens | `design/tokens.json` → `src/generated-tokens.css` | CSS custom properties (single source of truth) |
| Global base | `src/app.css` | Reset, shell layout, shared base classes |
| Semantic utilities | `src/utilities.css` | Reusable layout/surface/text helpers |
| Component-local | `src/lib/components/**/*.css` | Per-component scoped rules |

**Observed pattern:** Tailwind v4 utility classes appear mostly in the `ui/` component wrappers (via `tailwind-variants`). Stage, cast, and overlay components use raw CSS custom properties almost exclusively. The two worlds currently coexist without a firm boundary.

**Global token variables** are imported at the top of `app.css` via `@import "./generated-tokens.css"` and are available everywhere. Tailwind utilities reference some of these via a `@theme inline {}` block that maps them to Tailwind's semantic names (`--primary`, `--secondary`, etc.).

**Recommended pattern per CLAUDE.md:** component CSS files paired with each `.svelte` file; shared base classes in `app.css`; state classes use `.is-` prefix.

## 2. Token System

### Source of Truth
**File:** `design/tokens.json`

**Generator:** `scripts/generate-tokens.ts` — run with `bun run generate:tokens`

**Outputs (both generated — do not hand-edit):**
- `control-panel/src/generated-tokens.css` — full token set for the Svelte app
- `public/tokens.css` — subset for OBS overlay HTML files

### Token Categories

| Category | Examples |
|---|---|
| **Colors** | `--red` (#ff4d6a), `--cyan` (#00d4e8), `--purple` (#500df5) |
| **Surface neutrals** | `--black`, `--black-card`, `--black-elevated`, `--grey`, `--grey-dim` |
| **Health states** | `--hp-healthy`, `--hp-injured`, `--hp-critical` + `-dim` variants |
| **Typography** | `--font-display` (Bebas Neue), `--font-mono` (JetBrains Mono), `--font-ui` (system-ui), `--font-script` (Dancing Script) |
| **Spacing** | `--space-half` (2px) through `--space-12` (48px) — 4px base unit |
| **Shadows** | `--shadow-card`, `--shadow-red`, `--shadow-cyan`, `--shadow-purple` (pixel-art offset style) |
| **Motion** | `--t-fast` (150ms), `--t-normal` (250ms), `--t-spring` (cubic-bezier spring) |
| **Z-index** | `--z-sidebar-backdrop` (30) → `--z-nav` (50) → `--z-modal` (70) |

### Naming Convention
Raw tokens are single-purpose (e.g., `--red`). No explicit semantic alias layer (e.g., `--color-danger`) — semantic meaning is carried by class names (`.is-critical`, `.hp-fill`), not token names. The `@theme inline` block in `app.css` maps some tokens to shadcn-compatible Tailwind semantic names (`--primary`, `--secondary`, `--destructive`) for the `ui/` component layer.

### shadcn Compatibility
A `@theme inline {}` block in `app.css` already maps `--red` → `--destructive`, `--black-card` → `--card`, etc. The `ui/` components are therefore shadcn-compatible in structure, though they were not generated by the shadcn-svelte CLI.

## 3. Component Architecture

### Directory Structure

```
control-panel/src/lib/components/
├── stage/          ← Operator UI (HP management, dice, setup)
├── ui/             ← Reusable headless + tailwind-variants wrappers (18 subdirs)
├── overlays/       ← OBS overlay Svelte components
└── cast/dm/        ← DM & player-facing components
```

### `ui/` — Low-Level Primitives (18 Components)

Built with **bits-ui v2** (headless behaviour) + **tailwind-variants** (variant styling) + `cn()` (clsx + tailwind-merge). This is the shadcn-svelte pattern applied manually.

Notable members:

| Component | Key feature |
|---|---|
| `button/` | `buttonVariants` — size (sm/default/lg/icon), variant (default/destructive/outline/ghost/link) |
| `badge/` | `badgeVariants` — default/secondary/destructive/outline |
| `stepper/` | Numeric ±/value/+ with hold-to-repeat; sizes sm/lg |
| `dialog/` | 6 sub-parts: trigger, overlay, content, title, footer, description |
| `alert-dialog/` | Confirmation variant of dialog |
| `condition-pill/` | Status effect badges for character cards |
| `tooltip/` | bits-ui tooltip primitives (underused — P2 adoption opportunity) |

### `stage/` — Application-Specific Components

These are fully app-specific — they directly consume `$characters` store, emit REST calls, and handle Socket.io side effects. They are **not** meant to be reused generically.

| Component | Responsibility |
|---|---|
| `CharacterCard.svelte` | HP, conditions, resources per character |
| `DiceRoller.svelte` | D&D dice roller with character selector |
| `CharacterCreationForm.svelte` | Multi-field character wizard |
| `MultiSelect.svelte` | Custom `<select multiple>` replacement |
| `PhotoSourcePicker.svelte` | Avatar upload/URL selection |

### `cast/dm/` — DM Route Components

| Component | Responsibility |
|---|---|
| `InitiativeStrip.svelte` | Two-mode initiative tracker (pre-combat rolls + combat strip) |
| `SessionCard.svelte` | Per-character session action card |
| `SessionBar.svelte` | Global session state bar |

### Pattern: Low-Level Usage in Routes

Currently the `stage/` components are consumed **directly in route pages** — e.g., `/live/characters/+page.svelte` renders a list of `<CharacterCard>` components. The `ui/` primitives are consumed **inside** `stage/` components rather than directly in routes. This is the right layering direction but not consistently enforced.

### Especially Good Pattern — `stepper/`

The `Stepper.svelte` component encapsulates hold-to-repeat logic (250ms delay, 80ms repeat interval) cleanly behind a `size` prop. It is reused in both `CharacterCreationForm` and `DiceRoller`, demonstrating the correct abstraction level for this codebase.

### Especially Messy Pattern — `CharacterCard`

`CharacterCard.svelte` is doing too much: HP bar, conditions, resources, rest actions, bulk select, collapse toggle, and anime.js damage flash — all in one file with ~16KB of paired CSS. This is a prime extraction candidate (see §7).

## 4. UI Library Usage

### bits-ui v2

Used as a **behaviour provider**, not a styled component. The project wraps bits-ui primitives in its own styled shells:

```
bits-ui primitive (headless)
  └── project wrapper (styled, in ui/)
        └── stage/cast component (app-specific)
              └── route page
```

**Import pattern (observed in `dialog/`):**
```js
import { Dialog as DialogPrimitive } from "bits-ui";
// then export named sub-parts:
// DialogTrigger, DialogContent, DialogOverlay, DialogTitle...
```

This matches the shadcn-svelte "compound component via named exports" pattern.

### tailwind-variants

Used only in `ui/` layer. Pattern:

```js
import { tv } from "tailwind-variants";
import { cn } from "$lib/utils.js";

export const buttonVariants = tv({
  base: "...",              // shared classes
  variants: { size: {...}, variant: {...} },
  defaultVariants: { size: "default", variant: "default" }
});

// In template:
<button class={cn(buttonVariants({ size, variant }), className)}>
```

**Not used in `stage/` or `cast/` components** — those rely entirely on CSS custom properties + BEM-style class names. This is the biggest inconsistency in the current codebase.

### Would Wrapper Components Fit the Style?

Yes. The `ui/` layer already demonstrates this (e.g., `Button.svelte` wraps `<button>` with variant logic). Creating higher-order wrappers like `AppNav.svelte`, `CharacterHPBar.svelte`, or `ConditionList.svelte` would follow the exact same pattern already established.

## 5. Best-Practice Pattern for This Repo

### Recommended Folder Structure for New Wrappers

```
control-panel/src/lib/components/
├── ui/                 ← Keep: low-level primitives (bits-ui + tailwind-variants)
├── stage/              ← Keep: operator-facing app components
│   ├── CharacterCard/
│   │   ├── CharacterCard.svelte       ← orchestrator (thin)
│   │   ├── CharacterCard.css          ← card shell styles only
│   │   ├── HPBar.svelte               ← extracted sub-component
│   │   ├── HPBar.css
│   │   ├── ConditionList.svelte       ← extracted sub-component
│   │   └── ResourcePips.svelte        ← extracted sub-component
│   └── DiceRoller/
│       ├── DiceRoller.svelte
│       ├── DiceRoller.css
│       └── DiceButton.svelte          ← extracted sub-component
├── cast/dm/            ← Keep: DM/player-facing components
└── shared/             ← NEW: cross-context components (used by both stage + cast)
    └── CharacterAvatar.svelte         ← photo + initials fallback (reused today in 3+ places)
```

### Where Things Should Live

| What | Where |
|---|---|
| Raw color/space/motion values | `design/tokens.json` (generates CSS vars) |
| Shared base classes | `src/app.css` (`.card-base`, `.btn-base`, `.label-caps`) |
| Semantic layout helpers | `src/utilities.css` (`.row`, `.stack`, `.surface-card`) |
| Low-level variant styles | `ui/` + tailwind-variants |
| App-specific shell styles | `ComponentName.css` paired with `.svelte` |
| Complex state CSS (conditions/animations) | `overlays/overlays.css` stays as-is |

### What to Keep vs. Hide in Wrappers

- **Keep `ui/` primitives hidden** — stage/cast components should import from `ui/`, not from bits-ui directly
- **Do not hide CSS custom properties** — they are the design system; keep them visible in component CSS files
- **Keep socket logic in route pages or top-level stage components** — do not push socket calls into extracted sub-components

### Data-Driven Rendering

The codebase already uses iterable rendering as the dominant pattern:

```svelte
<!-- from /live/characters/+page.svelte -->
{#each $characters as char (char.id)}
  <CharacterCard character={char} />
{/each}
```

New components should follow this: accept a typed data object as a prop, derive display state inside the component using `$derived`, and emit events upward rather than calling server APIs directly.

## 6. Concrete Examples

### `control-panel/src/lib/components/stage/CharacterCard.svelte` + `.css`
- Demonstrates the **paired .svelte/.css co-location** pattern used throughout
- Shows how `.is-critical`, `.is-selected` BEM-ish state classes are applied from reactive props
- Uses `animate()` from animejs for damage flash — the canonical approach for programmatic animation
- **Problem:** single file doing too much — HP bar, conditions, resources, collapse, bulk select, animation

### `control-panel/src/lib/components/ui/button/index.js`
- Shows the **compound component export pattern**: `export { Button, buttonVariants }`
- `buttonVariants` is exported separately so callers can compose class lists without rendering the element
- Clean separation between variant logic (JS/TS) and template (Svelte)

### `control-panel/src/app.css`
- Shows the **`@theme inline {}` bridge** between CSS custom properties and Tailwind semantic tokens
- Shows `!important` font enforcement to prevent Tailwind base styles from overriding design tokens
- Shows where `.card-base`, `.btn-base`, `.label-caps` shared utility classes live

### `control-panel/src/lib/stores/socket.js`
- Single Socket.io client instance exported as a module singleton
- `characters` and `lastRoll` are `writable` stores — subscribed in components via `$characters` rune
- Demonstrates the firm rule: **components never mutate stores directly**; they send REST → server broadcasts → store updates

### `control-panel/src/lib/components/overlays/overlays.css`
- Pure CSS animation definitions for all condition effects (poisoned, burning, frozen, stunned, etc.)
- Shows how `data-char-id` attribute DOM targeting works alongside CSS class conditions
- `.heal-flash` keyframe — one-shot brightness burst, no JS required

## 7. Suggested Extraction Targets

### Target 1: `HPBar.svelte` — extract from `CharacterCard`

**Current state:** HP bar markup, color logic, and transition CSS are embedded in `CharacterCard.svelte/css` (~16KB combined).

**Extract to:** `stage/CharacterCard/HPBar.svelte` + `HPBar.css`

**Props:** `{ current, max, isCollapsed }`

**Why:** The HP bar is the most-reused visual — it appears in `CharacterCard`, `OverlayHP`, and `OverlayCharacterFocus`. Extracting it eliminates duplication and localises the `.is-critical` threshold logic.

---

### Target 2: `CharacterAvatar.svelte` — new shared component

**Current state:** The photo + initials fallback pattern (photo URL → display; missing → first letter of name in colored circle) is duplicated across `CharacterCard`, `OverlayDice`, `OverlayHP`, and `InitiativeStrip`.

**Extract to:** `shared/CharacterAvatar.svelte`

**Props:** `{ photoUrl, name, size?: 'sm' | 'md' | 'lg' }`

**Why:** Smallest possible interface, no socket dependency, purely presentational. Perfect first extraction.

---

### Target 3: `ConditionList.svelte` — extract from `CharacterCard`

**Current state:** Condition badges (condition-pill components), add/remove handlers, and the `@apply` logic for condition filter classes are all in `CharacterCard`.

**Extract to:** `stage/CharacterCard/ConditionList.svelte`

**Props:** `{ conditions: string[], characterId: string }`

**Emits:** `add(condition)`, `remove(condition)` — parent handles server call

**Why:** Conditions appear in `CharacterCard` (editable) and overlay components (read-only). A shared component with a `readonly` prop would serve both contexts. The `overlays.css` condition filter classes would still be applied at the overlay route level, not inside this component.
